# Source-App Internal NLE Timeline Architecture Plan - Owner Map & Compatibility Risks

본 문서는 Source-App 기반 내부 NLE 타임라인 구조 변경 계획에 따른 소유 디렉토리/파일 구조 및 내재 호환성 리스크 분석서입니다.

---

## 1. Owner-File Map (상태/기능 소유 모델)

### [1] project payloads
- **역할**: 프로젝트 파일의 전체 스키마 명세 및 디스크 영속성 저장 구조
- **소유 파일**:
  - `core/project/project_format.py` (스키마 버전 제어, 비디오 헤더 리프레시)
  - `core/project/project_io.py` (msgpack/json 바이너리 인코딩, zlib 압축, 아토믹 IO, LRU 캐시)
  - `core/project/project_manager.py` (프로젝트 생성, 변경 상태 관리, 직렬화 게이트웨이)

### [2] media assets
- **역할**: 미디어 트랙 및 비디오/오디오 클립 메타데이터
- **소유 파일**:
  - `core/project/project_assets.py` (자막 및 미디어 파일 트랙 맵, 세그먼트 메타 데이터 관리)
  - `core/project/project_format.py` (클립 단위의 `timeline_start_sec`, `timeline_end_sec`, `fps`, `frame_count` 수집 및 검증)

### [3] editor segments
- **역할**: 편집기 내에서 표시되는 자막 세그먼트와 STT/LLM 상태 정보
- **소유 파일**:
  - `ui/editor/editor_segments_runtime_cache.py` (런타임 자막 세그먼트 고속 인메모리 캐싱)
  - `ui/editor/editor_save_manager.py` (편집 상태의 비동기 저장 및 디스크 자막 에셋 정렬)
  - `ui/editor/editor_project_open_native.py` (로컬 디스크에서 세그먼트를 로드하여 타임라인 위젯에 로드)

### [4] roughcut candidates
- **역할**: 러프컷 후보 세그먼트 분류 및 챕터화 결과물
- **소유 파일**:
  - `core/project/project_roughcut_store.py` (러프컷 중간 세그먼트 분류 구조화 및 디스크 콤팩션)
  - `core/roughcut/models.py` (`RoughCutSegment`, `SemanticChunk` 등의 데이터 모델 선언)

### [5] cut-boundary seeds
- **역할**: 컷 경계 검출 및 FPS 보정 캐시 데이터
- **소유 파일**:
  - `core/cut_boundary.py` (컷 경계 정규화 및 프로비저널 컷 관리)
  - `core/cut_boundary_fps.py` (Avfoundation/Ffmpeg probe 결과의 FPS 노멀라이즈 오버라이드)

### [6] render plans
- **역할**: EDL 생성을 포함한 러프컷 비디오 재결합/렌더링 시나리오
- **소유 파일**:
  - `core/roughcut/edl_generator.py` (EDL 텍스트 포맷 빌드)
  - `core/roughcut/render_executor.py` (FFMPEG 파이프라인 호출)

### [7] sidecars
- **역할**: 프로젝트 저장 폴더 하위에 매칭되는 외부 자막 및 분석 캐시 정보
- **소유 파일**:
  - `core/project/project_assets.py` (외부 `.srt` 및 캐시 sidecar 리딩)

### [8] timeline canvas state
- **역할**: 타임라인 UI 상에서의 zoom, scroll, hover, active segment
- **소유 파일**:
  - `ui/timeline/timeline_canvas.py` (PyQt 뷰 및 제스처 핸들링)
  - `core/project/project_context.py` (런타임 UI 정보(`pps`, `zoom`, `scroll_x`) 소거)

### [9] save/reopen behavior
- **역할**: 편집 내용 저장 시점의 캐시 플러시 및 복원
- **소유 파일**:
  - `ui/editor/editor_save_manager.py` (세그먼트 정합성 및 백업)

## 2. Hidden Compatibility Risks (내재적 데이터 호환성 리스크)

1. **인메모리 캐시 소실 리스크**:
   - `core/project/project_io.py`의 `_project_payload_for_disk` 호출 시 `_external_subtitle_segments_cache` 등의 런타임 키가 소거됩니다. 만약 메모리 상의 프로젝트 인스턴스를 Deep Copy 없이 참조 변환하여 저장하게 되면, 실행 중이던 타임라인의 메모리 캐시 정보가 날아가버려 이후 편집 동작 시 데이터 싱크 오류를 야기합니다.
2. **이종 FPS 복합 타임라인 리스크**:
   - 멀티클립 구조에서 각 비디오 클립의 원래 FPS(예: 29.97fps, 23.976fps)와 프로젝트 기준의 `primary_fps`(예: 30.0fps)가 불일치할 때, 클립 리스트 정보에서 개별 FPS 정보를 타임라인 글로벌 눈금과 병합하지 않으면 특정 위치에서 렌더링 프레임 수가 불일치하는 현상이 나옵니다.
3. **물리 Sidecar 파일 불일치**:
   - `.aissproj` 파일은 아토믹하게 자동 백업 저장되나, 매핑되는 외부 `.assets/` 하위의 자막 파일 이름 등이 임의로 변경되거나 예외가 발생할 경우, 자막 정보 복구 로직이 제대로 폴백되지 않고 초기화(공백)되어 데이터를 날릴 가능성이 존재합니다.

## 3. [Critical Risk] Cut Boundary Point vs Clip Boundary Span 혼용 위험

### [1] 개념적 본질 차이
- **Cut Boundary Point (컷 경계 지점)**:
  - 타임라인 전체의 **글로벌 절대 시간축(Timeline Global Absolute Time)** 상에 위치한 단일 컷 분기 지점의 프레임 인덱스 혹은 타임스탬프(Point).
- **Clip Boundary Span (클립 범위 스팬)**:
  - 개별 원본 미디어 비디오 클립 리소스의 **로컬 시간축(Clip Local Absolute Time)** 상에서의 시작점과 끝점 범위(In-Out points, duration)를 뜻함.

### [2] 혼용 및 오계산 시 유발되는 구체적 위험
- **오프셋 누적으로 인한 싱크 드리프트(Sync Drift)**:
  - 프로젝트 글로벌 FPS(예: 30fps)와 클립 원본 FPS(예: 29.97fps)가 다를 때, `Cut Boundary Point` 프레임을 로컬 클립의 `Clip Boundary Span` 값에 그대로 대입하여 프레임 가산을 하면 타임라인 후반부로 갈수록 자막 싱크와 실제 컷 편집점이 벌어져 싱크가 완전히 깨지는 현상이 발생합니다.
- **프레임 단위 버림/반올림 오차로 인한 Flash Frame(잔상)**:
  - 초(sec) 단위를 프레임 단위로 양방향 매핑할 때, 스냅(`snap_sec_to_frame`) 오차 보정이 양 축 간에 정확히 이루어지지 않으면 렌더링 빌드(`edl_generator.py`) 시 인접 접합부에서 1프레임 미만의 껌벅임(검은 화면 또는 이전 컷 잔상)이 내보내기 결과물에 혼입될 수 있습니다.
- **FFMPEG 물리적 절단 범위 에러**:
  - 드래그를 통해 글로벌 `Cut Boundary Point`를 강제로 늘렸으나 로컬 클립의 `Clip Boundary Span` 유효 범위(물리적 파일의 duration)를 넘겨서 동기화 연산이 수행되는 경우, 렌더러 파이프라인에서 존재하지 않는 영상 영역에 읽기를 시도하여 FFMPEG `Invalid Data` 예외 오류가 나며 러프컷 렌더링이 전체 실패하는 치명적 락(Lock)이 초래됩니다.
